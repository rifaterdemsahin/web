To run this analysis directly from a Logic App, you can use the Azure Logic Apps and Azure Functions combination. Here's a step-by-step guide to achieve this:

  • Create an Azure Function: Write an Azure Function that performs the analysis.

  • Trigger the Azure Function from the Logic App: Use an HTTP trigger or a timer trigger in the Logic App to call the Azure Function.

  • Output the results in the Logic App: Handle the results from the Azure Function in the Logic App workflow.

Step 1: Create an Azure Function

First, create an Azure Function that performs the Logic App runs analysis.

Create a PowerShell Azure Function

  • Create the Azure Function App:

  • Navigate to the Azure portal.

  • Create a new Function App.

  • Choose the PowerShell Core runtime stack.

  • Create a PowerShell Function:

  • In the Function App, create a new function using the HTTP trigger template.

  • Write the Function Code:

  • Replace the template code with the following PowerShell code:

using namespace System.Net

Input bindings are passed in via param block.

param($Request, $TriggerMetadata)

Authenticate to Azure

Connect-AzAccount -Identity

Define variables

$resourceGroupName = "YourResourceGroupName"
$logicAppName = "YourLogicAppName"

Fetch the runs for the Logic App in the specified resource group

$startDate = (Get-Date).Date # Start from the beginning of today
$endDate = $startDate.AddDays(1) # End at the beginning of tomorrow

Get the Logic App runs for the specified date range

$logicAppRuns = Get-AzLogicAppRunHistory -ResourceGroupName $resourceGroupName -Name $logicAppName -StartTime $startDate -EndTime $endDate

Initialize counters

$successCount = 0
$failureCount = 0
$skippedCount = 0

Loop through the runs and count the statuses

foreach ($run in $logicAppRuns) {
switch ($run.Status) {
"Succeeded" { $successCount++ }
"Failed" { $failureCount++ }
"Skipped" { $skippedCount++ }
}
}

Prepare the response

$response = @{
Date = (Get-Date -Format 'yyyy-MM-dd')
SucceededRuns = $successCount
FailedRuns = $failureCount
SkippedRuns = $skippedCount
}

Output the results

$responseBody = $response | ConvertTo-Json
Write-Output $responseBody

Create the response object with status code.

$Response = [HttpResponseMessage]::new([HttpStatusCode]::OK)
$Response.Content = [System.Net.Http.StringContent]::new($responseBody, [System.Text.Encoding]::UTF8, "application/json")
return $Response

Step 2: Create the Logic App

  • Create a Logic App:

  • Navigate to the Azure portal.

  • Create a new Logic App.

  • Add an HTTP Action:

  • In the Logic App Designer, add a new HTTP action.

  • Set the HTTP action to perform a GET request to the Azure Function URL.

Step 3: Configure the HTTP Action

  • HTTP Action Configuration:

  • Set the method to GET.

  • Set the URL to the Azure Function's URL (you can find this in the Function App's "Get function URL" option).

  • Parse the JSON Response:

  • Add a new action to parse the JSON response from the Azure Function.

  • Choose the "Parse JSON" action.

  • Set the content to the body of the HTTP response.

  • Define the schema based on the expected response from the Azure Function. You can generate the schema using the following JSON:
    json { "Date": "string", "SucceededRuns": "integer", "FailedRuns": "integer", "SkippedRuns": "integer" }

  • Add Actions to Handle the Results:

  • Add actions to handle the parsed results, such as sending an email or logging the results to a storage account.

Example Logic App Workflow

  • HTTP Action:

  • Method: GET

  • URL: https://<your-function-app>.azurewebsites.net/api/<your-function>?code=<your-function-key>

  • Parse JSON:

  • Content: @body('HTTP')

  • Schema:
    json { "type": "object", "properties": { "Date": { "type": "string" }, "SucceededRuns": { "type": "integer" }, "FailedRuns": { "type": "integer" }, "SkippedRuns": { "type": "integer" } } }

  • Log Results or Send Notification:

  • Use the parsed data to log results, send an email notification, or take any other action as needed.

By following these steps, you can create a workflow in Azure Logic Apps that analyzes daily Logic App runs by invoking an Azure Function. The function will perform the analysis and return the results, which can then be processed further in the Logic App.

The error message Assembly with same name is already loaded typically occurs when there is an attempt to load a module or assembly that has already been loaded into the current session. This can happen if multiple modules are trying to load the same assembly or if there's an explicit Import-Module that conflicts with an implicit load.

Step-by-Step Solution to Resolve Assembly Conflicts

  • Remove Redundant Imports:
    Ensure that modules are imported only once and there are no conflicting imports.

  • Use Specific Imports:
    Instead of importing the entire Az module, import only the necessary submodules.

  • Clear Loaded Modules:
    Clear any loaded modules if re-importing is necessary.

The error message indicates that the managed identity authentication is failing, possibly due to issues with the managed identity configuration or permissions. To resolve this, ensure that the managed identity is correctly set up and has the necessary permissions to access Azure resources.

Steps to Fix Managed Identity Authentication Issues

  • Verify Managed Identity Configuration:
    Ensure that the managed identity is enabled for the Function App.

  • Assign Necessary Roles:
    Assign the necessary roles to the managed identity to access Azure resources, such as the Logic App.

  • Use Azure PowerShell with Service Principal (as an alternative to managed identity).

Step 1: Verify Managed Identity Configuration

  • Navigate to Your Function App:

  • In the Azure portal, go to your Function App.

  • Enable Managed Identity:

  • Under the "Settings" section, select "Identity".

  • Ensure that the "System assigned" managed identity is set to "On".

Step 2: Assign Necessary Roles

  • Assign Roles in the Resource Group:

  • Go to the resource group containing your Logic App.

  • Select "Access control (IAM)".

  • Click "Add" and then "Add role assignment".

  • Assign the "Logic App Contributor" role to the managed identity of your Function App.

  • Verify Role Assignment:

  • Ensure that the managed identity has the necessary roles assigned.

Step 3: Use Azure PowerShell with Service Principal (Alternative)

If managed identity continues to fail, you can use a service principal for authentication. Here's how to set it up:

  • Create a Service Principal:

  • Use the Azure CLI to create a service principal and get the client ID, secret, and tenant ID:
    sh az ad sp create-for-rbac --name "<your-service-principal-name>" --role contributor --scopes /subscriptions/<your-subscription-id>

  • Update Azure Function to Use Service Principal:

  • Store the client ID, client secret, and tenant ID in Azure Key Vault or as application settings in the Function App.

  • Modify the function code to authenticate using the service principal.
