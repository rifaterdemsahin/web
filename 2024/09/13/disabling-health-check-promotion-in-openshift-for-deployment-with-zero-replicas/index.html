Disabling Health Check Promotion in OpenShift for Deployment with Zero Replicas

When working with OpenShift, there are times when you need to deploy a module without triggering the usual health check promotion pipeline. This might be necessary for various reasons, such as performing maintenance or updating the application with zero downtime. In this blog post, we’ll go over how to temporarily disable the production module health check promotion pipeline to facilitate a deployment with zero replicas.

Why Disable Health Check Promotion?

By default, OpenShift uses health checks to ensure that each module is running correctly before promoting it to production. However, there are scenarios where you may want to bypass these checks temporarily, such as when:

  • Deploying updates or patches that don't require immediate scaling.

  • Performing maintenance tasks that necessitate starting with zero replicas.

  • Preparing an environment for a controlled deployment where immediate promotion isn't desired.

Steps to Disable Health Check Promotion

  • Access Your OpenShift Console:
    Begin by logging into your OpenShift console. Ensure you have the appropriate permissions to modify deployment configurations.

  • Navigate to the Deployment Configuration:
    Find the deployment configuration (DC) of the module you wish to modify. This can typically be found under the "Workloads" section in the OpenShift console.

  • Edit the Deployment Configuration:
    Click on the module's deployment configuration to edit it. You'll need to adjust the settings to temporarily disable health checks.

  • Set the Replicas to Zero:
    Adjust the replicas field in your deployment configuration to 0. This ensures that no pods are running during the initial stage of deployment.

  • Disable Health Check Promotion:
    Locate the health check settings in the deployment configuration. These are often found under the "Health Checks" tab or section. Temporarily disable the readiness and liveness probes to prevent the deployment pipeline from promoting the module based on health checks.

  • Apply Changes:
    Save and apply your changes to the deployment configuration. This will update the configuration and disable the health check promotion for this module.

  • Deploy Your Module:
    With health checks disabled and replicas set to zero, you can now deploy your module without triggering the health check promotion pipeline. This setup allows for controlled updates or maintenance without the usual checks and balances.

  • Re-enable Health Checks After Deployment:
    Once your deployment or maintenance task is complete, remember to re-enable the health checks and adjust the replicas back to the desired number to resume normal operations.

Explanation of Health Checks:

  • Liveness Probe (livenessProbe):

  • Purpose: Checks if the application is still running. If the probe fails, Kubernetes will restart the container.

  • Configuration:

httpGet: Makes an HTTP GET request to the /healthz endpoint on port 8080.

  • initialDelaySeconds: Waits 10 seconds after the container starts before performing the first check.

  • periodSeconds: Checks every 10 seconds.

  • timeoutSeconds: Waits up to 5 seconds for a response before timing out.

  • failureThreshold: If the probe fails 3 times consecutively, the container is considered unhealthy and will be restarted.

  • successThreshold: The number of consecutive successes required to mark the container as healthy (only 1 is needed here).

  • Readiness Probe (readinessProbe):

  • Purpose: Checks if the application is ready to accept traffic. If the probe fails, the container is marked as not ready, and traffic is not routed to it.

  • Configuration:

httpGet: Makes an HTTP GET request to the /readiness endpoint on port 8080.

  • initialDelaySeconds: Waits 5 seconds after the container starts before performing the first check.

  • periodSeconds: Checks every 5 seconds.

  • timeoutSeconds: Waits up to 3 seconds for a response before timing out.

  • failureThreshold: If the probe fails 3 times consecutively, the container is marked as not ready.

  • successThreshold: The number of consecutive successes required to mark the container as ready (only 1 is needed here).

What to Adjust if Needed:

  • Path: Ensure that the /healthz and /readiness endpoints are correctly implemented in your application. Adjust these paths as needed.

  • Port: Make sure port 8080 is the correct port where your application is running. Update it if necessary.

  • Timings: Adjust initialDelaySeconds, periodSeconds, timeoutSeconds, failureThreshold, and successThreshold based on your application’s startup and runtime behavior.

By defining these probes, Kubernetes can automatically manage the availability and readiness of your application, improving its resilience and reliability.

  • Log in to your OpenShift Web Console using your credentials.

  • Navigate to the project (namespace) where your deployment (e.g., golang-ex-new) is located. You can switch projects from the top-left corner dropdown.

2. Navigate to the Workloads Section

  • In the left-hand navigation pane, click on "Workloads".

  • Click on "Deployments" to see the list of deployments in the selected project.

3. Find and Select Your Deployment

  • Find your deployment (e.g., golang-ex-new) in the list of deployments.

  • Click on the deployment name to open its details page.

4. View or Edit the Liveness Probe

  • Once you are on the deployment's details page, click on the "Actions" dropdown menu at the top right corner.

  • Select "Edit Health Checks" from the dropdown menu. This option allows you to view and configure the liveness and readiness probes for your containers. Alternatively, you might directly see an option for "Health Checks" or "Edit Health Checks" in the navigation tabs of the deployment's details page, depending on the version of OpenShift you are using.

5. Spot the Liveness Probe Configuration

  • In the Health Checks configuration window:

  • You will see sections for Liveness Probe, Readiness Probe, and possibly Startup Probe.

  • The Liveness Probe section displays the current configuration if it is already set. Look for fields such as:

Type (HTTP, Command, or TCP)

  • Timeout (seconds)

  • Period (seconds)

  • Failure Threshold

  • To edit the liveness probe:

  • After updating the liveness probe, OpenShift will automatically restart the deployment pods to apply the new settings.
